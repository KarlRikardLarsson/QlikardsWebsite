Does Pre-Sorting Speed Up GROUP BY in Qlik?

April 10, 2026 · 7 min read

If you aggregate large RESIDENT tables with GROUP BY, you might have wondered whether the order of the data matters. Does pre-sorting with ORDER BY before grouping actually help? I ran a proper benchmark to find out — and the answer is consistently yes, though how much depends heavily on the granularity of what you're grouping.

The two approaches

The standard way to aggregate a RESIDENT table is straightforward — one pass, let Qlik figure out the grouping internally: 

// Approach A — GROUP BY only
AggResult:
LOAD
    Dimension1,
    Sum(Amount) AS TotalAmount
RESIDENT Transactions
GROUP BY Dimension1;

DROP TABLE AggResult;

The alternative is to first sort the source table by the grouping key, then aggregate from the sorted copy: 

// Approach B — Pre-sort, then GROUP BY
Tmp_Sorted:
LOAD
    Dimension1,
    Amount
RESIDENT Transactions
ORDER BY Dimension1;

AggResult:
LOAD
    Dimension1,
    Sum(Amount) AS TotalAmount
RESIDENT Tmp_Sorted
GROUP BY Dimension1;

DROP TABLE AggResult;
DROP TABLE Tmp_Sorted;

Approach B does more work on the surface — it creates an intermediate table and runs two passes. But the hypothesis is that a sorted source makes the GROUP BY phase significantly cheaper, more than offsetting the sorting cost.

Test setup

All tests ran on Qlik Cloud using RESIDENT loads against a generated table of 5 million rows. Each scenario was run five times and results are the average across all iterations.

The base table was generated with four dimensions of varying cardinality: 

Transactions:
LOAD
    RecNo()                             AS TransID,
    Chr(65 + Mod(RecNo()-1, 26))        AS Dimension1,  // ~26 distinct values
    Chr(65 + Mod(RecNo()-1, 26))
      & Chr(65 + Mod(Floor((RecNo()-1)/26), 26))        AS Dimension2,  // ~676 combinations
    Hash128(RecNo())                    AS Dimension4,  // ~unique per row
    Round(Rand() * 10000, 0.01)         AS Amount
AutoGenerate 5000000;

Four scenarios were tested, each with both approaches:

  • 1 — Low granularity, single field: GROUP BY Dimension1 (~26 distinct values)
  • 2 — High granularity, single field: GROUP BY Dimension4 (~unique per row via Hash128)
  • 3 — Low + Low, two fields: GROUP BY Dimension1, Dimension2 (~676 combinations)
  • 4 — Low + High, two fields: GROUP BY Dimension1, Dimension4

Timing was captured using a logging subroutine that records start and end timestamps per test iteration: 

SUB RunTests

    // ── Scenario 1A — Low gran, GROUP BY only ──────────────
    LET vStart = Timestamp(Now(), 'hh:mm:ss');
    Agg: LOAD Dimension1, Sum(Amount) AS Total RESIDENT Transactions GROUP BY Dimension1;
    DROP TABLE Agg;
    CALL sLog('1A Low gran — GROUP BY only', '$(vStart)', Timestamp(Now(), 'hh:mm:ss'));

    // ── Scenario 1B — Low gran, Pre-sort + GROUP BY ────────
    LET vStart = Timestamp(Now(), 'hh:mm:ss');
    Tmp_Sorted: LOAD Dimension1, Amount RESIDENT Transactions ORDER BY Dimension1;
    Agg: LOAD Dimension1, Sum(Amount) AS Total RESIDENT Tmp_Sorted GROUP BY Dimension1;
    DROP TABLE Agg; DROP TABLE Tmp_Sorted;
    CALL sLog('1B Low gran — Pre-sort + GROUP BY', '$(vStart)', Timestamp(Now(), 'hh:mm:ss'));

    // ── Scenario 2A — High gran, GROUP BY only ─────────────
    LET vStart = Timestamp(Now(), 'hh:mm:ss');
    Agg: LOAD Dimension4, Sum(Amount) AS Total RESIDENT Transactions GROUP BY Dimension4;
    DROP TABLE Agg;
    CALL sLog('2A High gran — GROUP BY only', '$(vStart)', Timestamp(Now(), 'hh:mm:ss'));

    // ── Scenario 2B — High gran, Pre-sort + GROUP BY ───────
    LET vStart = Timestamp(Now(), 'hh:mm:ss');
    Tmp_Sorted: LOAD Dimension4, Amount RESIDENT Transactions ORDER BY Dimension4;
    Agg: LOAD Dimension4, Sum(Amount) AS Total RESIDENT Tmp_Sorted GROUP BY Dimension4;
    DROP TABLE Agg; DROP TABLE Tmp_Sorted;
    CALL sLog('2B High gran — Pre-sort + GROUP BY', '$(vStart)', Timestamp(Now(), 'hh:mm:ss'));

    // ── Scenarios 3 and 4 follow the same pattern ──────────

END SUB

The results

Pre-sorting outperformed GROUP BY only in every single scenario. The margin varied, but the direction was always the same:

Scenario GROUP BY only Pre-sort + GROUP BY Difference
Low granularity, single field 0:00:07 0:00:05 −2s  (29% faster)
High granularity, single field 0:00:32 0:00:26 −6s  (19% faster)
Low + Low, two fields 0:00:09 0:00:08 −1s  (11% faster)
Low + High, two fields 0:00:33 0:00:26 −7s  (21% faster)
These are the combined times for the full pipeline — including the ORDER BY pass in approach B. The pre-sort approach does more total work and still wins.

Why does it help?

When Qlik processes a GROUP BY without any prior sorting, it has no guarantees about where matching keys appear in the data. It builds a hash table of partial aggregates — every time it encounters a row, it hashes the grouping key, finds the right bucket, and accumulates the value there. For 5 million rows, that's 5 million hash operations and 5 million bucket lookups.

If the data is already sorted by the grouping key, a completely different strategy becomes possible: sequential processing. Qlik only needs to track the current group. When the key changes, the current group is finalized and a new one starts. No hash table, no random memory access, no bucket collisions — just a linear scan with minimal state.

This is why the absolute gains are largest on high-granularity fields, but the proportional gain is actually largest on low-granularity fields. With only 26 distinct values, the hash table is tiny and collisions are frequent — every row almost certainly lands in a bucket that's already been touched, making cache performance poor. With sorted data, you make at most 26 group transitions across 5 million rows. The difference in access pattern is dramatic even if the absolute numbers are smaller.

The two-field scenarios

The multi-field results follow the same pattern. Low + Low is faster across the board because 676 distinct combinations is still a manageable hash table. Low + High is the worst case — the second field is effectively unique, so the combined key is nearly unique per row, and the GROUP BY is doing almost no real aggregation. You're paying the full cost of a hash table with ~5 million entries to produce ~5 million output rows.

Pre-sorting by both fields (ORDER BY Dimension1, Dimension4) converts that into a linear scan and saves 7 seconds — the largest absolute gain in the test.

If your GROUP BY produces nearly as many rows as the source table, that's a signal your grouping key has very high cardinality. Pre-sorting will give you the biggest gains here.

The cost of the extra pass

Approach B runs an additional RESIDENT load just to sort — that's not free. The fact that it's still faster overall means the sort cost is being more than recovered in the GROUP BY phase. But there's a memory implication: you briefly hold two copies of the table in RAM — the original and the sorted copy — before the DROP cleans one up.

Because Tmp_Sorted only loads the fields needed for the aggregation — not every column in the source table — its memory footprint is smaller than the original. On a wide transaction table with many fields, that difference can be significant. Even so, on very large datasets where memory is tight, the extra table is still a cost. Always DROP it as soon as the aggregation is done.

When to use it

  • High-cardinality GROUP BY keys — the hash table pressure is worst here, and sorting gives you the cleanest win.
  • Multi-field GROUP BY with at least one high-cardinality field — same reason, larger combined key space.
  • Recurring aggregations — if the same RESIDENT table is grouped by several different combinations in sequence, consider sorting once by the most selective field first and reusing the sorted table.

For low-granularity aggregations on smaller tables, the gains are real but small enough that the added code complexity may not be worth it. The 2-second saving on a 7-second operation is 29% — meaningful in a loop or a nightly job, less so in a one-off reload.

The pattern itself is simple enough that there's rarely a reason not to use it when you know your GROUP BY key has high cardinality. Sort, aggregate, drop the sorted table. Three lines of overhead for consistent gains across every scenario tested.
