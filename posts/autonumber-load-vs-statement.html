Autonumber During LOAD vs After — Which Is Faster?

April 13, 2026 · 5 min read

I always use Autonumber() for my key fields. Converting a long string key like 'Company1|TXN-000123' to a small integer saves a significant amount of memory — Qlik stores the integer symbol once and references it everywhere, while a string gets duplicated across every row in every table that uses it. For large models with millions of rows across multiple fact tables, this adds up fast.

There are two ways to apply it: inline as a function inside the LOAD statement, or as a standalone Autonumber statement after the LOAD using a wildcard. I benchmarked both to see if it matters where you put it.

Why a prefix on key fields?

The % prefix on the field name is a Qlik convention for key fields. It's not required, but it signals intent to anyone reading the script — and, as we'll see at the end, it makes it easy to apply rules to all keys at once using a wildcard.

Approach 1: Autonumber inline in the LOAD

The most common pattern — call Autonumber() directly on each key field inside the LOAD: 

LET vTestStart = timestamp(now(),'hh:mm:ss.fff');

Tmp_Sorted:
LOAD
    *,
    Autonumber('Company1|' & Dimension1) AS %KEY_1,
    Autonumber('Company1|' & Dimension2) AS %KEY_2,
    Autonumber('Company1|' & Dimension3) AS %KEY_3,
    Autonumber('Company1|' & Dimension4) AS %KEY_4
RESIDENT Transactions
;

DROP TABLE Tmp_Sorted;

CALL sLog('Autonumber During LOAD','$(vTestStart)',timestamp(now(),'hh:mm:ss.fff'));

Approach 2: Autonumber statement with a wildcard

Load the string keys first, then apply autonumbering in a separate pass using the standalone Autonumber statement with a wildcard: 

LET vTestStart = timestamp(now(),'hh:mm:ss.fff');

Tmp_Sorted:
LOAD
    *,
    'Company2|' & Dimension1 AS %KEY_1,
    'Company2|' & Dimension2 AS %KEY_2,
    'Company2|' & Dimension3 AS %KEY_3,
    'Company2|' & Dimension4 AS %KEY_4
RESIDENT Transactions
;

Autonumber '%KEY_*';

DROP TABLE Tmp_Sorted;

CALL sLog('Autonumber %KEY_*','$(vTestStart)',timestamp(now(),'hh:mm:ss.fff'));

Results

Both approaches produce identical output — the difference is only in how long they take. Run over a large transaction table with multiple key fields:

  • Autonumber during LOAD: 0:00:17
  • Autonumber with wildcard after LOAD: 0:00:24

The inline approach was about 30% faster. The reason is that the wildcard statement requires a second pass over the data — it loads the full string keys first, then rewrites them all as integers in a separate step. The inline function does both in one pass, row by row, as the LOAD runs.

The wildcard approach is still useful when you're working with a table that was already loaded without autonumber — for instance, when retrofitting keys to a table loaded by someone else's script. But if you're writing the LOAD yourself, do it inline.

Bonus: exclude key fields from Smart Search

Since all key fields follow the %KEY_* naming convention, you can exclude them from Smart Search indexing with one line: 

search exclude '%KEY*';

Smart Search indexes every field by default, including keys — but keys are integers that carry no meaning to the end user. Excluding them reduces index size and keeps search results clean. The % prefix makes the wildcard trivial to write: one rule covers every key in the entire model, no matter how many you add later.
