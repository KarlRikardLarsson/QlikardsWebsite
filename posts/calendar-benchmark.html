Calendar Scripts — Which One Is Fastest?

April 7, 2026 · 6 min read

There are several ways to build a master calendar in Qlik. Most developers pick one early on and stick with it — but they're not all equal. Inspired by Rob Wunderlich's calendar scripts, I ran a benchmark across three common approaches to see how much the choice actually matters.

The three methods

Each method generates the same output — a DIM_Calendar with date, year, month, day, week, weekday, quarter and a YTD flag — but they differ in how they find the min/max date range.

1. Min/max via RESIDENT

The most explicit approach. A temp table aggregates Min() and Max() from the fact table, stores the result in variables, then uses those to drive an AUTOGENERATE date sequence. 

TMP1_Calendar:
LOAD
    Min(TransactionDate) as MinDate,
    Max(TransactionDate) as MaxDate
RESIDENT Transactions;

LET vMinDate = Peek('MinDate', 0, 'TMP1_Calendar');
LET vMaxDate = Peek('MaxDate', 0, 'TMP1_Calendar');
DROP TABLE TMP1_Calendar;

TMP2_Calendar:
LOAD
    Date($(vMinDate) + RowNo() - 1) as TransactionDate
AUTOGENERATE $(vMaxDate) - $(vMinDate) + 1;

DIM_Calendar:
LOAD
    TransactionDate,
    Year(TransactionDate)                      as Year,
    Month(TransactionDate)                     as Month,
    Day(TransactionDate)                       as Day,
    Week(TransactionDate)                      as Week,
    WeekDay(TransactionDate)                   as WeekDay,
    'Q' & Ceil(Month(TransactionDate) / 3)     as Quarter,
    YearToDate(TransactionDate)                as YTDFlag
RESIDENT TMP2_Calendar;

DROP TABLE TMP2_Calendar;

2. FieldValueCount + WHILE loop

Skips the RESIDENT aggregation entirely. Uses FieldValue() and FieldValueCount() to read min/max directly from Qlik's internal symbol table, then generates the date range with a WHILE loop using IterNo(). The calendar fields are computed in a preceding load on top. 

DIM_Calendar:
LOAD
    TempDate                               as TransactionDate,
    Year(TempDate)                         as Year,
    Month(TempDate)                        as Month,
    Day(TempDate)                          as Day,
    Week(TempDate)                         as Week,
    WeekDay(TempDate)                      as WeekDay,
    'Q' & Ceil(Month(TempDate) / 3)        as Quarter,
    YearToDate(TempDate)                   as YTDFlag;
LOAD
    date(mindate + IterNo()) AS TempDate,
    maxdate
WHILE mindate + IterNo() <= maxdate;
LOAD
    min(FieldValue('TransactionDate', recno())) - 1 as mindate,
    max(FieldValue('TransactionDate', recno()))     as maxdate
AUTOGENERATE FieldValueCount('TransactionDate');

3. FieldValue + AUTOGENERATE (WHILE-less)

A cleaner variant of method 2. Reads min/max via FieldValue() into a temp table, stores them as variables, then uses a straightforward AUTOGENERATE — no WHILE loop. 

Calendar_MinMax:
LOAD
    Num(Min(_DateScan)) AS MinDate,
    Num(Max(_DateScan)) AS MaxDate
;
LOAD
    FieldValue('TransactionDate', IterNo()) AS _DateScan
AUTOGENERATE 1
WHILE NOT IsNull(FieldValue('TransactionDate', IterNo()));

LET vCal_MinDate = Peek('MinDate', 0, 'Calendar_MinMax');
LET vCal_MaxDate = Peek('MaxDate', 0, 'Calendar_MinMax');
DROP TABLE Calendar_MinMax;

DIM_Calendar:
LOAD
    TempDate                               as TransactionDate,
    Year(TempDate)                         as Year,
    Month(TempDate)                        as Month,
    Day(TempDate)                          as Day,
    Week(TempDate)                         as Week,
    WeekDay(TempDate)                      as WeekDay,
    'Q' & Ceil(Month(TempDate) / 3)        as Quarter,
    YearToDate(TempDate)                   as YTDFlag;
LOAD
    Date($(vCal_MinDate) + RecNo() - 1) AS TempDate
AUTOGENERATE ($(vCal_MaxDate) - $(vCal_MinDate) + 1);

The test setup

Each method was run 5 times against the same dataset using a logging subroutine that captures start and end timestamps. Average execution time was measured per method. The dataset was generated with AUTOGENERATE and includes a TransactionDate field spanning 5 years back from today with 10 000 000 rows. 

SUB sLog(vMethod, vStart, vEnd)
    LOG_Results:
    LOAD
        '$(vMethod)'                                     as Method,
        Time('$(vEnd)') - Time('$(vStart)')              as Time
    AUTOGENERATE 1;
END SUB

Both FieldValue-based methods are essentially instant. The RESIDENT min/max approach averaged 14 seconds — every run.

Why is RESIDENT so much slower?

The RESIDENT approach scans every row in the fact table to compute Min() and Max(). On a large dataset, that's a full table pass just to get two numbers.

The FieldValue() methods bypass this entirely. Qlik's associative engine already maintains an internal symbol table of every distinct value per field — it's what powers the green/white/grey selection states in list boxes. Reading min and max from that index costs almost nothing, regardless of how many rows are in the fact table.

This is the same principle as the ApplyMap existence check trick — FieldValue() reads from the symbol table, not the data rows. Once you internalise that distinction, a lot of performance optimisations start to make sense.

Which should you use?

Both FieldValue methods are equally fast here. The WHILE loop variant (method 2) is slightly more compact but harder to read at a glance. Method 3 — the AUTOGENERATE variant — separates the min/max lookup from the date generation, which is easier to follow and debug. Due to this, I personally prefer method 3.

Either way: stop using RESIDENT for calendar date ranges. The performance difference is real, it costs nothing to switch, and the code is no more complex.
