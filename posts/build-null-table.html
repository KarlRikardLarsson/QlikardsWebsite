BuildNullTable — Handle Missing Dimension Keys Automatically

April 3, 2026 · 6 min read

Every data model has them: fact rows that reference a key that doesn't exist in the dimension table. Maybe it's dirty source data, a late-arriving dimension, or a deleted record that left orphaned transactions behind. Without handling, those rows show up as blank or NULL in every chart and table in the dashboard. Users notice. They ask questions. It erodes trust.

BuildNullTable is the subroutine I reach for to solve this cleanly and consistently — in one call per dimension table, with no manual field-by-field wiring.

The problem in plain terms

Say you have a Transactions fact table and a DIM_Customers dimension. Some transactions reference a %CustomerID that doesn't exist in DIM_Customers. Qlik can't associate those rows to any customer, so every customer attribute — name, region, segment — shows as - or blank.

The fix is to add a catch-all row to DIM_Customers: one row where the key field gets a special value that Qlik can associate to, and all attribute fields get a meaningful label like [Missing]. Users see something informative instead of nothing.

This post assumes your key fields are identified by a prefix — % by convention. If your model uses a different naming convention, just change the KeyPrefix parameter.

How it works

The subroutine loops through every field in the target dimension table using NoOfFields() and FieldName(). For each field it decides:

  • Key field (starts with the key prefix, e.g. %) → set to OtherSymbol (default +). This is Qlik's built-in "other" symbol — it associates correctly without creating false matches to real keys.
  • Attribute field → set to your chosen label, e.g. [Missing]. This is what users see in the dashboard.

It then builds a dynamic LOAD statement from those values and appends one row to the dimension table using AUTOGENERATE 1. One subroutine call, one null row, zero manual field maintenance.

Parameters

Parameter Description Example
TableName The dimension table to add the row to 'DIM_Customers'
OtherText The label shown for unmatched records '[Missing]'
KeyPrefix The character that identifies key fields '%'

Dependency

You must set OtherSymbol before calling the subroutine. This is the value Qlik uses internally for the "other" bucket — setting it yourself lets you control which symbol gets placed in the key fields of your null row. 

SET OtherSymbol = +;

The subroutine

SET OtherSymbol = +;

SUB BuildNullTable(TableName, OtherText, KeyPrefix)

    LET vLoadExpr   = '';
    LET vFieldCount = NoOfFields('$(TableName)');

    FOR i = 1 TO $(vFieldCount)

        LET vFieldName = FieldName($(i), '$(TableName)');

        // Key field → OtherSymbol (associable, no real match)
        IF Left('$(vFieldName)', 1) = '$(KeyPrefix)' THEN
            LET vFieldValue = '$(OtherSymbol)';
        ELSE
            // Attribute field → OtherText (shown in the dashboard)
            LET vFieldValue = '$(OtherText)';
        ENDIF

        // Build up the field expression for the LOAD statement
        IF Len('$(vLoadExpr)') > 0 THEN
            LET vLoadExpr = '$(vLoadExpr),'
                & Chr(10) & '    '
                & Chr(39) & '$(vFieldValue)' & Chr(39)
                & ' AS [$(vFieldName)]';
        ELSE
            LET vLoadExpr = '    '
                & Chr(39) & '$(vFieldValue)' & Chr(39)
                & ' AS [$(vFieldName)]';
        ENDIF

    NEXT i

    TRACE BuildNullTable: Adding null row to [$(TableName)] with label: $(OtherText);

    [$(TableName)]:
    LOAD
    $(vLoadExpr)
    AUTOGENERATE 1;

END SUB;

Calling it

One line per dimension table. Run this after each dimension is fully loaded — before you build any fact associations that depend on it. 

CALL BuildNullTable('DIM_CostCentre', '[Missing]', '%');
CALL BuildNullTable('DIM_Customers',  '[Missing]', '%');
CALL BuildNullTable('DIM_Projects',   '[Missing]', '%');
CALL BuildNullTable('DIM_Accounts',   '[Missing]', '%');
CALL BuildNullTable('DIM_Products',   '[Missing]', '%');
Only call this on dimension tables — never on fact or transaction tables. Appending a null row to a fact table will distort your aggregations.

What you get in the dashboard

Before BuildNullTable, unmatched fact rows produce blank cells and break totals. After, they all group neatly under [Missing] — visible, filterable, and honest about the data quality problem rather than silently hiding it.

That's also useful analytically. A filter on [Missing] instantly shows you the scope of the data quality issue: how many transactions are affected, which time periods, which amounts. It turns an invisible problem into a measurable one.

Why the key prefix convention matters

The whole subroutine depends on being able to tell key fields apart from attribute fields at script time. The % prefix convention — where %CustomerID is the key and CustomerName, Region etc. are attributes — makes that trivial. If your model doesn't use a consistent prefix, you'd need a different approach (an explicit list of key fields, for example). But if you're not already using a key prefix convention, this is a good reason to start.
