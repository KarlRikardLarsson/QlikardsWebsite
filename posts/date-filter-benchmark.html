Stop Using >= for Date Filters — Use WHERE EXISTS Instead

April 8, 2026 · 8 min read

If you've been writing WHERE TransactionDate >= '$(vDate)' to filter a QVD load, you're not alone — it's one of the most common patterns in Qlik scripts. It works. But it's not the fastest option, and there's a subtle trap hiding in that string comparison. A temp calendar with WHERE Exists() is both safer and quicker.

The problem with >=

On the surface, WHERE TransactionDate >= '1/1/2024' looks reasonable. In practice it compares the stored date value against a string literal, which means the result depends on your locale and how the date was formatted when it was written to the QVD. On a machine with a different regional setting — or after a QVD rebuild where the format changed — the filter silently returns the wrong rows.

The numerically safe version would be something like WHERE TransactionDate >= Num(Date#('1/1/2024','M/D/YYYY')), but that's verbose and easy to get wrong. There's a cleaner solution.

The four methods I tested

I benchmarked four approaches to loading a filtered date range from a QVD. All four return the same result — only the mechanics differ.

1. WHERE EXISTS with a temp calendar

Build a lightweight temporary table containing every date in the desired range using AUTOGENERATE, then use Exists() in the WHERE clause to filter the QVD load to only matching dates. Drop both tables when done. 

TMP_Calendar:
LOAD
    Date($(vMinDate) + RowNo() - 1) as TransactionDate
AUTOGENERATE $(vMaxDate) - $(vMinDate) + 1;

Transactions:
LOAD
    TransLineID,
    TransID,
    TransactionDate
    // ... other fields
FROM [lib://DataFiles/Transactions.qvd]
(qvd)
WHERE Exists(TransactionDate);

DROP TABLES Transactions, TMP_Calendar;

2. >= comparison

The classic hardcoded string comparison filter, the kind you see in most scripts.

Transactions:
LOAD
    TransLineID,
    TransID,
    TransactionDate
    // ... other fields
FROM [lib://DataFiles/Transactions.qvd]
(qvd)
WHERE TransactionDate >= '1/1/2024';

3. Inner join with a temp calendar

Load the full QVD first, then inner join a generated calendar onto it. This keeps only rows where TransactionDate exists in both tables. 

Transactions:
LOAD
    TransLineID,
    TransID,
    TransactionDate
    // ... other fields
FROM [lib://DataFiles/Transactions.qvd]
(qvd);

Inner join(Transactions)
LOAD
    Date($(vMinDate) + RowNo() - 1) as TransactionDate
AUTOGENERATE $(vMaxDate) - $(vMinDate) + 1;

DROP TABLES Transactions;

4. LEFT KEEP with a temp calendar

Similar to the inner join approach but uses LEFT KEEP instead — loads the full QVD, then trims it down to rows whose TransactionDate exists in the calendar. 

TMP_Calendar:
LOAD
    Date($(vMinDate) + RowNo() - 1) as TransactionDate
AUTOGENERATE $(vMaxDate) - $(vMinDate) + 1;

LEFT KEEP (TMP_Calendar)
Transactions:
LOAD
    TransLineID,
    TransID,
    TransactionDate
    // ... other fields
FROM [lib://DataFiles/Transactions.qvd]
(qvd);

DROP TABLES Transactions, TMP_Calendar;
Methods 3 and 4 both load the entire QVD into memory before filtering. That means they pay full I/O cost for every row regardless of how many you actually need — which matters a lot on large files.

Test setup

Each method was run 5 times against a 10 000 000-row transactions dataset. A logging subroutine captured start and end timestamps for each run, and the results were averaged. vMinDate and vMaxDate were pre-calculated variables covering the desired date range.

Why is WHERE EXISTS faster?

When Qlik evaluates WHERE Exists(TransactionDate), it uses the already-loaded TMP_Calendar table as a hash set. Each row from the QVD is checked against that set in constant time — it's a simple key lookup.

The >= approach evaluates a comparison expression for every single row as it streams through the QVD. That's 10 000 000 evaluations, each requiring the engine to interpret the value and the comparison operand.

Inner join and LEFT KEEP are worst off because they read the entire QVD into RAM first — all 10 million rows — and only then apply the filter. WHERE EXISTS filters during the read, so rows outside the date range are never materialised in memory at all.

The temp calendar approach also composes nicely with the fast calendar generation techniques covered in an earlier post. Use FieldValue() to find your min/max dates for free, then AUTOGENERATE the range, and filter with WHERE Exists(). The whole setup costs almost nothing extra.

The takeaway

Next time you reach for WHERE TransactionDate >= '$(vDate)', consider the temp calendar pattern instead. It's faster, it sidesteps locale-dependent string comparisons, and the intent is explicit — the calendar table makes it immediately obvious what date range you're loading.

The pattern looks like more code at first glance, but it's two short blocks with a clear purpose each. And on large QVDs, that 25% reduction in load time adds up.

Taking it further — the naming trap

WHERE EXISTS being the fastest isn't the whole story. That speed advantage only holds under one specific condition: the field name in the temp calendar must match the field name being loaded from the QVD. If they differ, you silently lose the optimized load and the performance gain disappears.

This is easy to miss because the script still runs correctly — the filter works, the row counts are right. The cost is invisible until you benchmark it.

What "optimized load" means here

Qlik's QVD engine has an optimized read mode: when a WHERE clause can be resolved entirely using the symbol tables of already-loaded fields, the engine skips full row parsing for excluded records. It's fast because it never materialises the rows it drops. WHERE Exists(TransactionDate) qualifies for this — but only when the field name in the Exists() check maps directly to a field that's already in memory with the same name as the field being loaded.

Three methods — the name matters

1. WHERE EXISTS — different field names (unoptimized)

The temp calendar is built with a field called TransactionDate2, but the QVD field is loaded as TransactionDate. They hold the same values — but the names don't match. Qlik can't use optimized mode and falls back to a row-by-row evaluation. 

TMP_Calendar:
LOAD
    Date($(vMinDate) + RowNo() - 1) as TransactionDate2
AUTOGENERATE $(vMaxDate) - $(vMinDate) + 1;

Transactions:
LOAD
    TransLineID,
    TransID,
    TransactionDate
    // ... other fields
FROM [lib://DataFiles/Transactions.qvd]
(qvd)
WHERE Exists(TransactionDate2, TransactionDate);

DROP TABLES Transactions, TMP_Calendar;

2. WHERE EXISTS — same field name via AS (optimized)

The fix is simple: use AS to alias the QVD field to match the calendar field name. Now both sides of the exists check share the same name, the engine recognises it as an optimized load, and the speed advantage returns. 

TMP_Calendar:
LOAD
    Date($(vMinDate) + RowNo() - 1) as TransactionDate2
AUTOGENERATE $(vMaxDate) - $(vMinDate) + 1;

Transactions:
LOAD
    TransLineID,
    TransID,
    TransactionDate as TransactionDate2   // renamed to match calendar
    // ... other fields
FROM [lib://DataFiles/Transactions.qvd]
(qvd)
WHERE Exists(TransactionDate2, TransactionDate);

DROP TABLES Transactions, TMP_Calendar;
If renaming the field downstream would break associations or chart expressions, you can always rename it back in a preceding load after the QVD read. The optimized read still fires on the QVD step — the preceding load runs in memory and costs very little.

3. ApplyMap — when you can't (or don't want to) rename

Sometimes renaming the field isn't practical — maybe it associates to multiple tables or you'd have to update a lot of downstream expressions. In that case, ApplyMap() is the right alternative. Build the temp calendar as a mapping table instead of a regular table, and filter on the returned value. 

TMP_Calendar:
Mapping LOAD
    Date($(vMinDate) + RowNo() - 1) as TransactionDate2,
    1
AUTOGENERATE $(vMaxDate) - $(vMinDate) + 1;

Transactions:
LOAD
    TransLineID,
    TransID,
    TransactionDate
    // ... other fields
FROM [lib://DataFiles/Transactions.qvd]
(qvd)
WHERE ApplyMap('TMP_Calendar', TransactionDate) = 1;

DROP TABLES Transactions;

Note that mapping tables are automatically dropped after a reload — you don't need an explicit DROP TABLE for them.

When the field names match (via AS), WHERE EXISTS is noticeably faster — 13 seconds versus 18–19 for the unoptimized alternatives. That's a ~30% improvement just from an alias. When the names differ, WHERE EXISTS and ApplyMap land at essentially the same speed — both are doing row-by-row evaluation with no optimized load.

Which should you use?

The decision comes down to whether you can make the field names match:

  • You can rename with AS → use WHERE EXISTS with matching names. It's the fastest option and the intent is clear.
  • You can't (or don't want to) rename → use ApplyMap. You won't get optimized load, but you get the same speed as unoptimized EXISTS with more flexibility: ApplyMap can handle mismatched names cleanly, and — unlike EXISTS — it can be extended to filter on multiple fields if needed.

Either way, the two-argument form of Exists() with different names is the worst of both worlds: it looks like an optimized load but isn't, and it offers no flexibility advantage over ApplyMap. Avoid it.
