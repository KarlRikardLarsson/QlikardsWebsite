Script Timers — Know Exactly How Long Every Section Takes

April 10, 2026 · 6 min read

A reload is slow. You open the reload log and see a wall of output — but nothing tells you which section is the culprit. You start adding manual TRACE statements with now(), doing mental arithmetic between timestamps. There's a much better way, and it's two subroutines you write once and never think about again.

StartTimer and EndTimer

Drop these two subs at the top of your script, or in a dedicated utilities tab: 

// -----------------------------------------------------------------------
// Call at the beginning and end of every section to track load times
// -----------------------------------------------------------------------

SUB StartTimer(timerName)
    TRACE ▶️ $(timerName);
    LET "vStart_$(timerName)" = now();
END SUB

SUB EndTimer(timerName)
    LET "vEnd_$(timerName)"      = now();
    LET "vDuration_$(timerName)" = Interval(vEnd_$(timerName) - vStart_$(timerName), 'hh:mm:ss');
    TRACE ⏱️ Execution time ($(timerName)): $(vDuration_$(timerName));

    TimerLog:
    LOAD
        '$(timerName)'                AS Section,
        '$(vStart_$(timerName))'      AS StartTime,
        '$(vEnd_$(timerName))'        AS EndTime,
        '$(vDuration_$(timerName))'   AS Duration
    AUTOGENERATE 1;

    DROP VARIABLES timerName, vStart_$(timerName), vEnd_$(timerName), vDuration_$(timerName);
END SUB

Basic usage — wrap any section with a matching pair of calls: 

CALL StartTimer('Load_Transactions');

Transactions:
LOAD ...
FROM [...] (qvd);

CALL EndTimer('Load_Transactions');

The reload log immediately becomes readable: 

▶️ Load_Transactions
⏱️ Execution time (Load_Transactions): 00:00:14

How it works

A few parts of this are worth understanding so you can adapt it.

Dynamic variable names. The double-quote syntax LET "vStart_$(timerName)" = now() creates a variable whose name is built from the parameter. If you call StartTimer('Transactions'), it creates a variable literally named vStart_Transactions. This means multiple timers can run without colliding — each one has its own uniquely named start and end variable.

Interval() for readable output. Subtracting two now() values gives a fractional day number. Interval(difference, 'hh:mm:ss') formats that into a human-readable duration. A section that takes 90 seconds shows as 00:01:30, not 0.001041667.

TimerLog auto-concatenation. Every call to EndTimer loads one row into a table named TimerLog. Because the table name is the same each time, Qlik auto-concatenates the rows — no explicit CONCATENATE needed. By the end of the reload, TimerLog holds a complete timing breakdown of every section that ran.

DROP VARIABLES. After logging, all four variables are dropped — the timer name and the three timestamps. This keeps the variable list clean and prevents stale values from interfering if you reuse the same section name across reloads.

The Execute_Sub upgrade

If your script is built around subroutines — each data domain in its own SUB — there's a wrapper that makes timing automatic with zero extra effort per section: 

SUB Execute_Sub(sub_routine)
    CALL StartTimer('$(sub_routine)');
    CALL $(sub_routine);
    CALL EndTimer('$(sub_routine)');
END SUB

CALL $(sub_routine) is a dynamic call — it resolves the variable to the actual subroutine name at runtime. Wrap any subroutine name in Execute_Sub and it gets timed automatically. Your main execution block becomes a single clean list: 

// ================================================================================
// MAIN EXECUTION
// ================================================================================

CALL Execute_Sub('Variables')
CALL Execute_Sub('Load_FX_Valutor')
CALL Execute_Sub('Load_DIM_KostnadsStälle')
CALL Execute_Sub('Load_DIM_Projekt')
CALL Execute_Sub('Load_DIM_Konto')
CALL Execute_Sub('Load_DIM_FinancialYear')
CALL Execute_Sub('Load_DIM_Artiklar')
CALL Execute_Sub('Load_DIM_Kunder')
CALL Execute_Sub('Load_FCT_Faktura')
CALL Execute_Sub('Load_FCT_Huvudbok')

Every section is automatically timed. Add a new subroutine and it gets timing for free — just add one CALL Execute_Sub line. Nothing else to remember.

The order in this block is your explicit load order. It doubles as documentation — anyone reading the script immediately sees the full sequence and can match it against the reload log.

What you get

By the end of every reload, you have two outputs:

  • The reload log — each section stamped with a start marker (▶️) and a duration (⏱️). Scannable at a glance without counting timestamps or doing arithmetic.
  • The TimerLog table — a proper in-model table with Section, StartTime, EndTime, and Duration columns. Build a monitoring sheet from it and you can compare reload performance across runs, spot regressions, and show stakeholders exactly where time is being spent.

The TimerLog table persists in the app's data model after each reload, which means you can slice it, sort it, and visualize it like any other table. A simple bar chart of Duration by Section immediately shows where the reload budget is going.

If you reload on a schedule and want to track timing trends over time, concatenate TimerLog into a QVD at the end of each reload. You can then load historical timing data and compare this week's section durations against last week's.
