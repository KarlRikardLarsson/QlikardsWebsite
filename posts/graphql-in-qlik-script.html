Calling GraphQL APIs from Qlik Script

April 13, 2026 · 7 min read

More and more SaaS systems expose their data through GraphQL instead of — or alongside — a traditional REST API. Qlik doesn't have a dedicated GraphQL connector, but it doesn't need one. GraphQL is just HTTP POST with a JSON body. The trick is safely constructing that body inside a Qlik script, where double quotes need special handling.

Why constructing the body is tricky

A GraphQL request body looks like this: 

{
  "query": "{ products { totalCount items { name productId supplierId } } }",
  "variables": { "supplierId": "S-10" }
}

The problem is that GraphQL queries contain double quotes — and in Qlik, when you embed a variable inside a string literal with $(vBody), a literal " in the variable value needs to be written as "" (Qlik's escape sequence for a double quote inside a single-quoted string). On top of that, the query string cannot contain raw newlines or tabs — those would break the JSON.

Doing this manually every time is error-prone and hard to read. The solution is a small utility subroutine that handles all of it in one place.

Create_Body

This subroutine takes a raw GraphQL query string, an optional variables object, and returns a properly formatted JSON body ready to use in a REST connector call: 

SUB Create_Body(_query, _variables, _body)

    /*
    Creates a valid GraphQL request body in JSON format.
    Strips illegal whitespace characters and escapes double quotes
    so the result can be safely embedded in a Qlik FROM clause.
    Inspired by: https://datafetcher.com/graphql-json-body-converter
    */

    IF Len(_variables) > 0 THEN

        // Query with variables
        LET _body = Replace(Replace(Replace(Replace('
        {
          "query": "$(_query)",
          "variables": $(_variables)
        }
        '
        , Chr(9),  ' ')          // Tab            → space
        , Chr(10), ' ')          // Newline         → space
        , Chr(13), ' ')          // Carriage return → space
        , '"', Chr(34)&Chr(34))  // "  →  ""  (Qlik escape for double quote)
        ;

    ELSE

        // Query without variables
        LET _body = Replace(Replace(Replace(Replace('
        {
          "query": "$(_query)"
        }
        '
        , Chr(9),  ' ')
        , Chr(10), ' ')
        , Chr(13), ' ')
        , '"', Chr(34)&Chr(34))
        ;

    ENDIF

ENDSUB

The four nested Replace calls handle all the illegal characters in sequence:

  • Chr(9) → space — removes tab characters that would break JSON
  • Chr(10) → space — removes Unix newlines
  • Chr(13) → space — removes Windows carriage returns
  • " → "" — escapes every double quote so it survives the $(vBody) expansion in the FROM clause
The whitespace replacements happen first deliberately. If you escaped the quotes first, the replacement strings would themselves contain quotes that the subsequent passes might mishandle. Order matters here.

Defining queries as subroutines

For each API endpoint you call, define a separate subroutine that sets the query string. This keeps your GraphQL queries readable — formatted with proper indentation — while keeping the main script clean: 

SUB Get_Products_Query(_query)
    /*
    Returns the query for the Products endpoint.
    The query can be copied directly from a GraphQL explorer (Postman, Insomnia, etc.)
    and pasted here without any manual formatting.
    */
    SET _query = '
    {
      products {
        totalCount
        items {
          name
          productId
          supplierId
        }
      }
    }
    ';
ENDSUB

One subroutine per endpoint. Each one is easy to update independently — paste a new version from Postman, save, done. No risk of accidentally breaking the body-formatting logic.

Calling the API

With the two utilities in place, loading data from a GraphQL endpoint is three lines of setup followed by a standard LOAD: 

// 1. Get the query string
CALL Get_Products_Query(vQuery);

// 2. Build the request body (no variables for this call)
CALL Create_Body(vQuery, '', vBody);

// 3. Load from the REST WITH CONNECTION 
WITH CONNECTION(
  HTTPHEADER "Authorization" "Bearer $(vAccessToken)",
  BODY "$(vBody)"
);

The field paths follow the GraphQL response structure. A response like { "data": { "products": { "items": [ {...}, {...} ] } } }

If you need to debug the body before hitting the API, add a TRACE $(vBody); after the Create_Body call. The formatted body prints to the reload log so you can inspect it directly.
